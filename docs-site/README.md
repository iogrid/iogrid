# docs.iogrid.org

Public documentation site for iogrid. Lives at https://docs.iogrid.org.

## Stack

- **[Astro](https://astro.build) 5** + **[Starlight](https://starlight.astro.build) 0.30**
- Static export (`astro build` → `dist/`), no runtime server
- **Pagefind** for client-side search (no third-party SaaS dependency)
- **MDX** content, **Shiki** code highlighting
- **OpenAPI** integration via `@astrojs/starlight-openapi` — renders
  `../proto/gen/openapi/iogrid.yaml` into `/api/reference/*` pages

### Why Starlight and not Mintlify

We evaluated both. Mintlify ships fast but ties the site to their hosted SaaS
(search, analytics, deploy). Starlight gives us:

- Pure static HTML output → drop it behind our own nginx (same as `marketing/`)
- MIT-licensed OSS, no vendor risk
- Built-in client-side search (Pagefind WASM) — no Algolia account, no
  recurring cost
- Native MDX + brand tokens from `../brand/tokens/colors.json`
- Astro's island architecture keeps the JS payload small (~30 KB total)

The choice can be revisited if Starlight hits a hard limit, but as of this
ship it covers every requirement on the EPIC.

## Develop

```bash
# From repo root
pnpm --filter @iogrid/docs-site install

# Fetch the latest OpenAPI spec (no-op when ../proto/gen/openapi/iogrid.yaml absent)
pnpm --filter @iogrid/docs-site openapi:fetch

# Run dev server at http://localhost:4321
pnpm --filter @iogrid/docs-site dev
```

The OpenAPI integration short-circuits to a placeholder sidebar group when
the spec is missing — the docs-site builds even on a fresh clone before the
SDK pipeline has run.

## Build

```bash
pnpm --filter @iogrid/docs-site openapi:fetch
node docs-site/scripts/gen-og.mjs
pnpm --filter @iogrid/docs-site build
pnpm --filter @iogrid/docs-site link-check
```

Output lands in `docs-site/dist/`. Serve it however you like — see the
`Dockerfile` for the canonical nginx-on-k8s recipe used at
`docs.iogrid.org`.

## Deploy

The shipped pipeline produces three artifacts:

1. **Static `dist/`** — upload to any static host (Cloudflare Pages, S3,
   GitHub Pages).
2. **OCI image** — built by the `docs-ci` workflow and pushed to
   `ghcr.io/iogrid/docs-site:<sha>` (in scope for Phase 0 k8s deploy behind
   `docs.iogrid.org`).
3. **Sitemap + robots.txt** — produced by Starlight at the standard URLs.

## Contribute

This site lives in the main iogrid monorepo, so the contribution flow is the
same as anywhere else:

1. Edit MDX files under `docs-site/src/content/docs/`.
2. Run `pnpm --filter @iogrid/docs-site dev` and visit the changed page.
3. Run `pnpm --filter @iogrid/docs-site typecheck` and `link-check`.
4. Submit a PR. The `docs-ci` workflow will rebuild, run link-check, and
   run Lighthouse against the major pages.

For changes to the OpenAPI-driven `/api/reference/` pages: those are
generated from `../proto/`. Edit the `.proto` files, run `make proto` from
the repo root, then build the docs.
