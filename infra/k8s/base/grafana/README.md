# Mothership Grafana dashboard provisioning for iogrid

iogrid does NOT run its own Grafana — it federates into the OpenOva
mothership Grafana running under `monitoring/grafana` on the bastion
k3s cluster (`45.151.123.50:6443`).

That Grafana already runs the kiwigrid `k8s-sidecar` container
watching every namespace for ConfigMaps with the label
`grafana_dashboard: "1"`. As long as iogrid's coordinator namespace
exists in the mothership cluster (handled by Flux at
`infra/k8s/base/namespaces/`), the sidecar auto-discovers our dashboards
and mounts them into Grafana with no manual API calls.

## Dashboards shipped by iogrid

The canonical JSON lives at
`coordinator/services/telemetry-svc/dashboards/` and is synced to
`infra/k8s/base/telemetry-svc/assets/dashboards/` via
`make sync-telemetry-assets`. Both copies must remain bit-identical
(CI enforces this).

The ConfigMap is generated by the `configMapGenerator` in
`infra/k8s/base/telemetry-svc/kustomization.yaml` and named
`iogrid-dashboards-<hash>`. Six dashboards:

| File | What it shows |
|------|---------------|
| `iogrid-overview.json` | Global posture — RPS, error rate, P99 across all services |
| `iogrid-services.json` | Per-service SLI rollups (proxy-gateway, build-gateway, identity-svc, …) |
| `iogrid-customers.json` | Per-customer usage panels |
| `iogrid-providers.json` | Per-provider earnings + workload mix |
| `iogrid-abuse.json` | Anti-abuse spike detector + top offenders |
| `iogrid-revenue.json` | GMV, payout queue depth, $GRID emission |

## How to verify provisioning landed

```bash
# 1. ConfigMap is in the cluster with the magic label.
k -n iogrid get cm -l grafana_dashboard=1

# 2. The mothership sidecar logged the pickup.
k -n monitoring logs deploy/grafana -c grafana-sc-dashboard | grep iogrid

# 3. The dashboard shows up in Grafana under the "iogrid" folder.
curl -fsS \
  -H "Authorization: Bearer $GRAFANA_API_KEY" \
  https://grafana.openova.io/api/search?folderIds=0\&query=iogrid \
  | jq '.[].title'
```

## Provisioning the iogrid Grafana folder

The dashboards land under the cluster-default "General" folder unless
we explicitly stamp a `grafana_folder` annotation. The
`folder-config.yaml` here creates a Folder Provisioning ConfigMap that
the sidecar picks up alongside the dashboards.

## Updating a dashboard

1. Edit the JSON under `coordinator/services/telemetry-svc/dashboards/`.
2. `make sync-telemetry-assets` (top-level Makefile) — copies the
   change into `infra/k8s/base/telemetry-svc/assets/dashboards/`.
3. Commit both files in one PR. Flux applies on merge, sidecar reloads
   within ~30 seconds.

Do NOT edit dashboards through the Grafana UI — they'll be reverted on
the next sidecar reconcile.
