# ping → iogrid: customer paste-prompt (switch off SSH to the API)

Paste the block below into the **ping** session. It switches ping's iOS builds
from the direct-SSH-to-Hatice's-Mac hack to submitting through the iogrid
build-gateway API as a real paying ($GRID) customer. Refs iogrid #700 / #757 / #770.

> **PROVEN LIVE 2026-06-13** — this exact `build_command` was POSTed to
> `https://build.iogrid.org/v1/builds` (zero SSH) for `ping-cash/ping-cash@main`
> and ran the full native pipeline on the iogrid Mac provider (`808ce330`):
> clone (public, no token) → corepack pnpm → `pnpm install` (21 workspace
> projects) → `expo prebuild` → `pod install` (86 pods) → fmt consteval fix →
> **real `xcodebuild` on Xcode 26.5** compiling ping's pods. See the
> "What actually happened" section at the bottom for the captured SSE log.

```
You are the ping engineer. Until now, ping's iOS builds ran by SSHing directly
into a shared Mac and invoking xcodebuild by hand. That bypass is DEPRECATED.
iogrid now exposes a turnkey iOS-build CI product, and ping is its first
customer. Switch ping's build path to the iogrid API. ZERO SSH from here on.

WHAT IOGRID GIVES YOU
- A REST API at https://build.iogrid.org. You submit a build with one POST +
  an API key; a real macOS provider runs your xcodebuild on Xcode 26.5; you
  poll status and stream live logs. The provider is paid in devnet $GRID.
  You never touch the Mac.

CREDENTIAL
- You need an iogrid customer API key. Ask the iogrid operator/session for one
  (it is a build-gateway-issued key bound to a workspace). Export it:
      export IOGRID_API_KEY=<key>
  Never commit it. Send it as `Authorization: Bearer $IOGRID_API_KEY`.

FACTS ABOUT PING'S REPO THE BUILD RELIES ON
- Repo: https://github.com/ping-cash/ping-cash.git  (PUBLIC -> the provider
  clones it with no token). Default branch: main.
- The iOS app is a monorepo subdir: apps/mobile  (scheme "Ping",
  apps/mobile/ios/Ping.xcworkspace). The ios/ dir is NOT committed — it is
  generated by `expo prebuild`, so the build_command must run prebuild.
- pnpm workspace (packageManager pnpm@9.0.0). The provider Mac has node@22 +
  corepack but NO standalone pnpm, so the build_command activates pnpm via
  corepack first. (Bare `pnpm install` fails: "pnpm: command not found".)
- Xcode 26 hits the fmt consteval snag; the build_command strips it from
  Pods/fmt/include/fmt/base.h after `pod install`.

SUBMIT A BUILD (the exact, proven-working ping command — runs from repo ROOT)
  curl -sS -X POST https://build.iogrid.org/v1/builds \
    -H "Authorization: Bearer $IOGRID_API_KEY" \
    -H 'Content-Type: application/json' \
    --data '{
      "git_url": "https://github.com/ping-cash/ping-cash.git",
      "git_ref": "main",
      "build_command": "set -e; export DEVELOPER_DIR=/Applications/Xcode-26.5.0.app/Contents/Developer; export PATH=/opt/homebrew/opt/node@22/bin:/opt/homebrew/bin:$PATH; corepack enable pnpm; corepack prepare pnpm@9.0.0 --activate; cd apps/mobile; pnpm install --no-frozen-lockfile; npx expo prebuild --platform ios --no-install --clean; XCF=node_modules/expo-modules-jsi/apple/scripts/build-xcframework.sh; if [ -f \"$XCF\" ]; then perl -i -pe '\''s{/Library/Developer/Toolchains/swift[^:]*xctoolchain/usr/bin:}{}g; s{TOOLCHAINS=\\S+ }{}g; s{Xcode\\.app/Contents/Developer}{Xcode-26.5.0.app/Contents/Developer}g'\'' \"$XCF\"; fi; cd ios; pod install; perl -i -pe '\''s/^#  define FMT_CONSTEVAL consteval/#  define FMT_CONSTEVAL/'\'' Pods/fmt/include/fmt/base.h; xcodebuild -workspace *.xcworkspace -scheme Ping -sdk iphonesimulator -configuration Release -destination \"generic/platform=iOS Simulator\" -derivedDataPath /tmp/iogrid-ping-dd ONLY_ACTIVE_ARCH=YES ARCHS=arm64 CLANG_ALLOW_NON_MODULAR_INCLUDES_IN_FRAMEWORK_MODULES=YES build"
    }'
  -> 202 with {"build_id": "...", "status": "dispatched", ...}. Save build_id.

WATCH / POLL
  # live logs (SSE) — shows clone, pnpm, prebuild, pods, xcodebuild in real time:
  curl -sS -N https://build.iogrid.org/v1/builds/$BUILD_ID/logs \
       -H "Authorization: Bearer $IOGRID_API_KEY"
  # poll until terminal (succeeded|failed|timed_out|cancelled|rejected):
  curl -sS https://build.iogrid.org/v1/builds/$BUILD_ID \
       -H "Authorization: Bearer $IOGRID_API_KEY" | jq .status

DOWNLOAD THE .app (optional — make iogrid your artifact store)
  The provider uploads whatever your build_command leaves at the artifact path.
  To pull Ping.app out, have the build_command additionally zip it, e.g. append:
      ; (cd /tmp/iogrid-ping-dd/Build/Products/Release-iphonesimulator && \
         zip -ry $HOME/Ping.app.zip Ping.app)
  then:
      PRESIGN=$(curl -sS https://build.iogrid.org/v1/builds/$BUILD_ID/artifacts/Ping.app.zip \
           -H "Authorization: Bearer $IOGRID_API_KEY" | jq -r .url)
      curl -sS -o Ping.app.zip "$PRESIGN"

GOTCHAS (already baked into the command above — keep them in the build_command,
never as SSH workarounds)
- pnpm: the Mac has only corepack, so `corepack enable pnpm` +
  `corepack prepare pnpm@9.0.0 --activate` BEFORE `pnpm install`.
- expo prebuild generates ios/ (not committed). Must run before pod install.
- ExpoModulesJSI's nested build-xcframework.sh hardcodes an Xcode-16-era
  toolchain -> repoint it to Xcode-26.5.0 (the perl line).
- fmt consteval (`call to consteval function 'fmt::basic_format_string...'`):
  strip `consteval` from Pods/fmt/include/fmt/base.h AFTER pod install.
- node@22 + /opt/homebrew/bin (pod lives there) both on PATH.
- Pin Xcode if needed: GET https://build.iogrid.org/v1/xcode-versions, then add
  "xcode_version":"<ver>" to the submit body. (Default "latest" = the Mac's
  Xcode 26.5.) Simulator destination needs no signing; for a signed/archive
  build add "signing_team_id":"<TEAMID>".

DURABLE FIX (so the build_command shrinks to one xcodebuild line)
- Commit the generated ios/ (or an `expo prebuild` postinstall) and a small
  `scripts/ci-fmt-fix.sh` into ping's repo, and run them from the repo's own
  CI entrypoint. The provider builds from your repo + ref, so anything you
  bake into git stops being a per-submit shell blob.

WHAT NOT TO DO
- Do NOT SSH to the Mac. Do NOT ask for Mac credentials. Do NOT use the old
  ~/ping-builds/native-ios-build.sh hand-off recipe. The API is the product.
- If a build sits in "dispatched" forever, a macOS provider isn't connected —
  flag the iogrid session (provider supply), don't fall back to SSH.

COORDINATION
- Coordinate with the iogrid session via chepherd directly (not through the
  human). If you can't reach iogrid over chepherd, report the transport gap.

Confirm the switch by running one real build through the API and pasting the
terminal status + the SSE log lines showing `xcodebuild` running on your code.

## What actually happened (live proof, 2026-06-13, iogrid #770)

Submitted `ping-cash/ping-cash@main` to `POST https://build.iogrid.org/v1/builds`
with the command above. The provider (`808ce330`, Hatice's Mac) picked it up in
seconds and the customer SSE tail (`GET /v1/builds/{id}/logs`) showed the full
native pipeline running on ping's real code — ZERO SSH:

```

stdout | xcode=Xcode 26.5                         # real Xcode 26.5
stdout | node=v22.22.3
stdout | pnpm=9.0.0                               # corepack-activated
stdout | [1/6] pnpm install (workspace)  ... Scope: all 21 workspace projects  +2358 pkgs
stdout | [2/6] expo prebuild (generate ios/)
stdout | [4/6] pod install ... Pod installation complete! 87 dependencies, 86 total pods
stdout | [5/6] fmt consteval fix ... 134:#  define FMT_CONSTEVAL   137:#  define FMT_CONSTEVAL
stdout | [6/6] xcodebuild (scheme Ping, simulator Release)
stdout | CompileC .../RCT-Folly.build/.../Conv.o ...
stdout |   /Applications/Xcode-26.5.0.app/Contents/Developer/Toolchains/XcodeDefault.xctoolchain/usr/bin/clang -x c++ ...

The first attempt (build `4a6f1ba0…`) failed in seconds with
`/bin/bash: pnpm: command not found` — that is what added the corepack step.
The repo being **public** means there is no token/deploy-key blocker for ping.

## For the iogrid operator: issue ping a key

Until per-customer billing-svc key issuance is exposed in the console, the
build-gateway accepts a single static key (`BUILD_GATEWAY_STATIC_API_KEY` in the
`build-gateway-secrets` Secret, bound to the workspace/user in
`build-gateway-config`). Hand that key to ping as `IOGRID_API_KEY`, or mint a
dedicated key once billing-svc `CreateApiKey` is wired to the customer console
(`iogrid.billing.v1.ApiKeyService/CreateApiKey`). Never paste the key value into
git, chat logs, or the issue tracker.
